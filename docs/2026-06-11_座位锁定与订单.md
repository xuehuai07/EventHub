# EventHub 阶段 3：座位锁定与订单

## 目标

完成用户从活动详情选择场次、查询实时库存、临时锁定固定座位或无座票、幂等创建订单、模拟支付、取消订单和查看订单的核心预约链路，并通过 Redis 原子操作与 MySQL 条件更新、唯一约束共同保证不重复锁定、不重复下单和不超卖。

## 范围

- 使用 Flyway `V3` 创建座位锁记录、订单、订单明细、支付记录和幂等记录表。
- 修正阶段 2 固定座位场次未生成 `eh_session_seat` 快照的问题。
- 固定座位场次按场馆座位等级关联票档并生成场次座位。
- 无座场次按票档和数量锁定库存。
- 实现场次实时可售座位、票档库存和锁定状态查询。
- 使用 Redis Lua 脚本原子锁定固定座位，支持多座位全成或全败。
- 使用 Redis Sorted Set 与 Lua 原子锁定无座票数量，并自动排除过期锁定。
- 座位锁默认有效期 10 分钟，支持用户主动释放。
- 创建订单要求 `Idempotency-Key`，重复请求返回同一订单。
- 创建订单时将 Redis 锁定转换为 MySQL 座位或库存占用。
- 使用 MySQL 条件更新和唯一约束作为最终防超卖保障。
- 实现订单状态机、模拟支付、待支付订单取消和超时关闭补偿。
- 用户端实现固定座位选择、无座票数量选择、锁定倒计时、订单确认、模拟支付、订单列表和订单详情。
- 管理端实现商家订单列表、订单详情和平台管理员订单查询。
- 增加固定座位争抢、无座库存争抢、重复下单和订单状态测试。
- 提供可重复执行的本地并发验收脚本。

## 非目标

- 本阶段不引入 RabbitMQ；订单延迟消息、Outbox 和死信处理留到阶段 4。
- 不生成正式电子票，不实现票券码和核销。
- 不实现真实支付平台、退款、优惠券、营销和发票。
- 不实现 WebSocket 实时推送；页面使用主动刷新和短周期查询更新状态。
- 不实现复杂座位图编辑器；沿用阶段 2 的区域、排号和座位号布局。
- 不支持已支付订单直接取消；退款状态流转留到后续阶段。
- 不允许修改已产生订单的活动、场次、票档和座位快照。

## 当前基础修正

阶段 2 已创建 `eh_session_seat` 表，但创建固定座位场次时没有生成场次座位快照。当前开发数据中：

- 场馆固定座位存在。
- 场次和票档存在。
- `eh_session_seat` 记录为空。

阶段 3 必须：

1. 在 `V3` 中为现有固定座位场次回填场次座位。
2. 按 `venue_seat.seat_grade = session_ticket_type.seat_grade` 关联票档。
3. 商家后续创建或修改固定座位场次时同步重建场次座位。
4. 提交活动审核前校验固定座位均能映射到有效票档。
5. 已发布或已有订单的场次禁止重建座位快照。

## 数据库影响

### 新增表

#### `eh_seat_lock_record`

- 锁定编号、用户、场次、票档、座位模式和数量。
- 状态：`ACTIVE`、`CONSUMED`、`RELEASED`、`EXPIRED`。
- 记录创建时间、过期时间、消费订单和更新时间。
- Redis 是临时锁定的实时来源，该表用于审计、补偿和问题排查。

#### `eh_seat_lock_item`

- 固定座位锁定明细。
- 保存锁定编号和 `session_seat_id`。
- 同一锁定内座位唯一。

#### `eh_ticket_order`

- 订单号、用户、商家、活动、场次、状态和金额。
- 状态：`PENDING_PAYMENT`、`PAID`、`CANCELLED`、`EXPIRED`。
- 保存支付截止时间、支付时间、取消时间和版本号。
- `(user_id, request_id)` 唯一，保证创建订单幂等。
- 订单号使用不可预测的时间有序随机编号，不直接暴露自增 ID。

#### `eh_ticket_order_item`

- 保存票档、场次座位、数量、单价和小计。
- 保存活动、场次、场馆、票档、区域、排号和座位号快照。
- 固定座位订单使用 `(session_seat_id)` 唯一约束，防止同一场次座位重复售卖。
- 无座订单的 `session_seat_id` 为 `NULL`，依赖票档库存条件更新防超卖。

#### `eh_payment_record`

- 保存模拟支付流水号、订单、金额、状态和支付时间。
- 同一订单只允许一个成功支付记录。

#### `eh_idempotency_record`

- 保存用户、接口范围、幂等键、请求摘要和关联资源。
- `(user_id, scope, idempotency_key)` 唯一。
- 相同键但请求摘要不同时返回幂等冲突。

### 现有表调整

- `eh_session_seat.status` 在本阶段只持久化 `AVAILABLE`、`SOLD`、`DISABLED`。
- `LOCKED` 是 Redis 临时状态，不写入 MySQL，避免锁过期与数据库状态不一致。
- `eh_session_ticket_type.available_stock` 表示尚未被订单占用的最终库存，不扣除 Redis 临时锁。
- 固定座位票档的 `total_stock` 必须与映射的有效场次座位数量一致。
- 商家修改场次票档时重建固定座位映射；已有订单后禁止修改。

## Redis 设计

### 固定座位锁

```text
eventhub:seat:lock:{sessionId}:{sessionSeatId}
```

- 值为锁定编号。
- TTL 默认 10 分钟。
- Lua 一次检查并写入全部座位 Key，任一座位已锁定则全部失败。
- 创建订单成功后删除锁 Key，不修改 MySQL 已售状态。
- 用户主动释放时只允许删除值等于当前锁定编号的 Key。

### 锁定详情

```text
eventhub:seat-lock:detail:{lockId}
```

- 保存用户、场次、票档、座位编号、数量和过期时间。
- TTL 与座位锁一致。
- 服务端不信任客户端传回的价格、数量和座位归属。

### 无座票临时锁

```text
eventhub:stock:locks:{ticketTypeId}
```

- 使用 Sorted Set。
- 每个临时占用单位保存一个 `lockId:index` 成员，score 为过期毫秒时间。
- Lua 在锁定前删除已过期成员，再使用 `ZCARD` 计算当前临时占用。
- 可锁数量为 `MySQL available_stock - Redis active_lock_count`。
- 锁定过期后无需恢复计数，下一次查询或锁定时自动清理过期成员。
- 创建订单成功后移除对应成员，同时 MySQL 条件扣减最终库存。

### 幂等进行中标记

```text
eventhub:idempotent:{userId}:{scope}:{key}
```

- 使用 `SET NX` 防止同一请求并发进入。
- 短 TTL 防止进程异常造成永久占用。
- MySQL 唯一约束是最终幂等保障，Redis 只减少并发重复执行。

## 锁定流程

### 固定座位

```text
校验用户、活动、场次和售票时间
-> 查询场次座位与票档
-> 校验座位均为 AVAILABLE 且属于同一场次
-> Lua 原子写入全部座位锁 Key
-> 写入锁定记录和座位明细
-> 写入锁定详情缓存
-> 返回 lockId、金额和 expiresAt
```

如果数据库锁定记录写入失败，服务端使用带值比较的 Lua 立即释放已写入的 Redis 座位锁。

### 无座票

```text
校验用户、活动、场次、票档和限购数量
-> 读取 MySQL available_stock
-> Lua 清理过期成员并原子判断剩余可锁数量
-> 写入锁定记录
-> 写入锁定详情缓存
-> 返回 lockId、金额和 expiresAt
```

如果数据库记录写入失败，立即删除该锁定编号对应的 Sorted Set 成员。

### 主动释放

- 只能释放当前用户自己的 `ACTIVE` 锁定。
- Redis 使用比较锁定编号的 Lua，不能删除其他用户后来创建的锁。
- 数据库状态更新为 `RELEASED`。
- 已消费、已释放或已过期的锁再次释放按幂等成功处理。

## 创建订单流程

客户端必须发送：

```text
Idempotency-Key: <UUID>
```

服务端流程：

1. 校验幂等键格式并计算请求摘要。
2. 查询 MySQL 幂等记录或已有 `(user_id, request_id)` 订单。
3. 获取 Redis 进行中标记。
4. 校验锁定属于当前用户、状态为 `ACTIVE` 且未过期。
5. 从数据库重新读取活动、场次、票档、价格和商家归属。
6. 固定座位使用条件更新将场次座位从 `AVAILABLE` 改为 `SOLD`。
7. 无座票使用 `available_stock >= quantity` 条件扣减库存。
8. 创建订单主表与订单明细快照。
9. 将锁定记录更新为 `CONSUMED` 并关联订单。
10. 写入幂等记录。
11. 事务提交后清理 Redis 锁定。
12. 返回同一订单结果。

任何数据库条件更新数量不符合预期时整体回滚，返回库存冲突。Redis 锁只提升并发体验，MySQL 条件更新和唯一约束负责最终不超卖。

## 订单状态机

本阶段允许：

```text
PENDING_PAYMENT -> PAID
PENDING_PAYMENT -> CANCELLED
PENDING_PAYMENT -> EXPIRED
```

约束：

- 创建订单后默认 15 分钟内支付。
- 模拟支付只能处理当前用户的 `PENDING_PAYMENT` 订单。
- 支付金额从数据库订单读取，不接受客户端金额。
- 取消待支付订单时恢复无座票库存或固定座位 `AVAILABLE` 状态。
- 超时关闭与取消使用同一个库存释放应用服务，保证幂等。
- 已支付订单不能取消，本阶段返回 `ORDER_STATUS_INVALID`。
- 支付和取消均使用订单版本或状态条件更新，防止并发支付与取消同时成功。

## 超时补偿

RabbitMQ 留到阶段 4，本阶段先提供可靠的数据库补偿基线：

- 每分钟扫描一小批支付截止时间已过的 `PENDING_PAYMENT` 订单。
- 使用状态条件更新或行锁确保同一订单只关闭一次。
- 关闭后恢复固定座位或无座票库存。
- 应用启动后会继续处理遗留超时订单。
- 阶段 4 接入 RabbitMQ 后，延迟消息作为主要触发方式，该补偿任务继续作为兜底。

## API 设计

### 座位与库存

```text
GET    /api/sessions/{sessionId}/availability
POST   /api/seat-locks
GET    /api/seat-locks/{lockId}
DELETE /api/seat-locks/{lockId}
```

`GET /api/sessions/{sessionId}/availability` 可匿名访问，但锁定接口必须使用 `USER_WEB` 普通用户身份。

固定座位响应包含：

- 场次座位 ID
- 区域、排号、座位号和座位等级
- 票档、价格
- `AVAILABLE`、`LOCKED`、`SOLD`、`DISABLED`

无座响应包含：

- 票档 ID、名称、价格
- MySQL 最终库存
- Redis 临时锁数量
- 当前可锁数量
- 每人限购数量

### 用户订单

```text
POST /api/orders
GET  /api/orders
GET  /api/orders/{orderId}
POST /api/orders/{orderId}/pay
POST /api/orders/{orderId}/cancel
```

- 创建、支付和取消写接口要求 `Idempotency-Key`。
- 用户只能读取和操作自己的订单。
- 列表支持状态筛选和分页。

### 管理端订单

```text
GET /api/merchant/orders
GET /api/merchant/orders/{orderId}
GET /api/admin/orders
GET /api/admin/orders/{orderId}
```

- 商家只能查看归属自己商家的订单。
- 管理员可以查询平台订单。
- 本阶段管理端只读，不允许人工修改订单状态。

## 权限与安全

- 查询可售座位允许匿名访问。
- 锁座、创建订单、支付、取消和用户订单查询只允许 `USER` 角色与 `USER_WEB` Token。
- 商家订单接口要求 `MERCHANT` 角色、有效商家状态和订单 `merchant_id` 归属。
- 管理员订单接口要求 `ADMIN` 角色。
- 客户端不能提交订单金额、商家编号或活动标题等可信业务字段。
- 锁定编号、订单号和支付流水号使用安全随机值。
- 锁座和创建订单接口增加用户与 IP 限流。
- 日志不得记录 Access Token、Refresh Token、完整支付流水或敏感用户数据。

## 风险

- Redis 锁定与 MySQL 订单事务属于跨存储操作，无法使用单一数据库事务；必须通过失败释放、最终数据库约束和补偿任务控制不一致。
- Redis 重启会丢失临时锁，但不能导致超卖；MySQL 条件更新和唯一约束仍会拒绝重复售卖。
- 固定座位映射依赖座位等级与票档等级一致，提交审核前必须严格校验。
- 创建订单后即占用最终库存，取消和超时必须可靠恢复。
- 支付与取消并发时只能有一个状态条件更新成功。
- 本阶段定时补偿不是精确延迟触发，订单可能在截止后短时间内才显示为过期；阶段 4 使用 RabbitMQ 改善及时性。
- 并发测试依赖 Docker Desktop 和真实 MySQL、Redis；当前 Docker Desktop 未运行，实施验收前必须启动。

## 回滚方案

- 停止锁座和订单 API，前端隐藏购票入口。
- 删除 Redis 中 `eventhub:seat:*`、`eventhub:stock:*` 和 `eventhub:idempotent:*` 临时键。
- 回滚代码不删除已创建订单数据。
- 需要结构补偿时新增 Flyway 修正迁移，不修改已执行的 `V3`。
- 已支付订单不得通过简单数据库回滚恢复库存，必须使用后续补偿脚本逐单核对。
- 开发环境无保留数据需求时可重建 Docker 数据卷。

### 并发验收

- 100 个请求争抢同一个固定座位，只允许一个成功。
- 多个用户一次选择多个固定座位，必须全部成功或全部失败。
- 1000 个请求争抢有限无座票，不允许锁定或下单数量超过库存。
- 同一用户使用相同 `Idempotency-Key` 重复创建订单，只产生一个订单。
- 相同幂等键但不同请求体返回 `IDEMPOTENCY_CONFLICT`。
- 支付与取消并发时只能成功一个状态转换。

### 业务验收

1. 用户查看场次实时座位状态。
2. 用户锁定固定座位并看到倒计时。
3. 第二个用户不能锁定同一座位。
4. 锁定过期或主动释放后，其他用户可以重新锁定。
5. 用户使用锁定创建待支付订单。
6. 重复提交创建订单返回同一订单。
7. 模拟支付后订单变为 `PAID`。
8. 取消待支付订单后库存恢复。
9. 超时待支付订单自动关闭并恢复库存。
10. 商家只能查看自己活动的订单。
11. 管理员可以查看平台订单。

## 验收标准

- 固定座位和无座票均可查询、锁定和创建订单。
- Redis Lua 保证临时锁定原子性。
- MySQL 条件更新和唯一约束保证最终不超卖。
- 同一幂等请求不会生成多笔订单或重复支付记录。
- 锁定过期、取消订单和超时订单能够正确释放库存。
- 用户不能访问他人订单，商家不能访问其他商家的订单。
- 用户端完成选座、订单确认、模拟支付和订单查询闭环。
- 管理端完成商家和管理员订单查询。
- 后端、两个前端和并发验收全部通过。
