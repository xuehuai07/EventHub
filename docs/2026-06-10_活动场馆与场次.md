# EventHub 阶段 2：活动、场馆与场次

## 目标

完成从商家创建活动、配置场馆与场次、提交平台审核，到管理员审核发布、普通用户浏览已发布活动的完整数据链路，为阶段 3 的座位锁定和订单创建提供稳定的活动、场次、票档与库存基础。

## 范围

- 使用 Flyway 创建活动分类、标签、场馆、场馆座位、活动、场次、票档和场次座位表。
- 实现活动状态机：`DRAFT`、`PENDING_REVIEW`、`PUBLISHED`、`REJECTED`、`OFF_SHELF`、`FINISHED`。
- 商家可以维护自己的场馆、活动草稿、场次和票档。
- 商家可以提交完整活动进入审核，驳回后可以修改并重新提交。
- 管理员可以查看待审核活动、通过、驳回和下架已发布活动。
- 普通用户可以分页、分类、按关键词和日期筛选已发布活动。
- 普通用户可以查看活动详情、场馆、场次和票档。
- 活动详情使用 Redis 缓存，活动修改或状态变化后主动失效。
- 用户端实现活动列表、筛选、详情和场次票档展示。
- 管理端实现商家活动列表、活动编辑、场馆管理、场次票档配置和管理员审核页面。
- 管理员可以创建和启停商家，并将已有用户绑定为商家员工。
- 商家可以配置固定座位场馆的区域、排号、座位号和座位等级。
- 用户可以查看场次票档及固定座位分布预览，但本阶段不能锁定或购买。
- 管理台工作台展示商家数、活动状态分布、已发布活动数和近期场次数。
- OpenAPI 作为前后端接口契约，并重新生成两个前端的 TypeScript 客户端。
- 提供本地商家演示账号的可选环境变量初始化能力。

## 非目标

- 本阶段不实现活动图片上传，封面只保存 URL；MinIO 留到后续阶段。
- 不实现座位临时锁定、并发扣减、订单、支付和票券。
- 不实现活动收藏、评价、推荐、全文搜索和 Elasticsearch。
- 不实现复杂富文本编辑器，活动介绍使用普通文本。
- 不实现商家认证材料上传和复杂审核流；管理员直接创建、启用或停用商家。
- 不实现周期性自动结束任务；`FINISHED` 状态先由业务查询判断或后续任务处理。
- 不允许商家自行发布、下架或修改其他商家的活动。

## 影响

### 数据库

新增以下表：

- `eh_activity_category`
- `eh_activity_tag`
- `eh_activity_tag_relation`
- `eh_venue`
- `eh_venue_seat`
- `eh_activity`
- `eh_activity_session`
- `eh_session_ticket_type`
- `eh_session_seat`

核心约束：

- 活动必须归属一个商家和一个分类。
- 场馆归属商家，商家只能引用自己的场馆。
- 活动使用乐观锁版本字段，避免编辑覆盖。
- 场次开始时间必须早于结束时间，售票时间必须覆盖有效销售区间。
- 票档金额使用分为单位的 `BIGINT`，禁止浮点金额。
- 同一场次票档名称唯一，票档库存不能为负。
- 场馆座位使用 `(venue_id, seat_code)` 唯一约束。
- 场次座位使用 `(session_id, venue_seat_id)` 唯一约束。
- 活动公开查询必须同时校验活动状态和场次有效性。
- 已执行的 Flyway 迁移不修改，本阶段新增 `V2` 迁移。

### 商家数据归属

- 商家接口通过当前用户查询 `eh_merchant_staff` 获得唯一 `merchant_id`。
- 只有状态为 `ACTIVE` 的商家和员工可以维护活动。
- 所有活动、场馆、场次和票档写操作必须在 SQL 或服务层同时校验 `merchant_id`，不能只校验 `MERCHANT` 角色。
- `ADMIN` 不默认拥有任何商家数据归属，只执行审核和平台操作。
- 本地环境可通过环境变量创建一个演示商家、商家员工和 `MERCHANT` 角色，不在仓库保存密码。

### 商家管理

- 管理员可以创建商家，商家初始状态为 `ACTIVE` 或由管理员明确指定。
- 管理员通过用户名或手机号查找已有用户并绑定为商家员工。
- 绑定成功后自动授予 `MERCHANT` 角色；解除最后一个商家归属时不自动删除历史角色，避免影响审计，角色回收留到权限管理阶段。
- 一个用户当前只允许归属一个商家，沿用 `eh_merchant_staff.user_id` 唯一约束。
- 停用商家后，其员工不能继续登录或调用商家业务接口，已发布活动是否下架由管理员单独操作。

### Redis

- `eventhub:activity:detail:{activityId}`：已发布活动详情，默认缓存 10 分钟。
- 查询未命中时读取 MySQL 并写入缓存。
- 草稿编辑、场次或票档变更、审核、驳回和下架后删除对应详情缓存。
- 不缓存商家草稿和管理员审核详情。
- 本阶段不引入分布式锁，数据库仍是活动状态和库存配置的最终可信来源。

### 分页契约

新增统一分页响应：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 12,
  "total": 0,
  "totalPages": 0
}
```

- 页码从 `1` 开始。
- `pageSize` 默认 `12`，最大 `100`。
- 公开活动默认按最近场次时间升序，再按活动编号倒序。
- 管理端列表支持状态和关键词筛选。

## 状态机

允许的状态转换：

```text
DRAFT -> PENDING_REVIEW
REJECTED -> PENDING_REVIEW
PENDING_REVIEW -> PUBLISHED
PENDING_REVIEW -> REJECTED
PUBLISHED -> OFF_SHELF
```

约束：

- `DRAFT` 和 `REJECTED` 可以编辑。
- `PENDING_REVIEW` 审核期间不可编辑。
- `PUBLISHED` 不允许直接修改影响售卖的数据；需要下架后在后续版本定义重新编辑流程。
- 提交审核前必须存在场馆、至少一个有效场次以及每个场次至少一个票档。
- 审核通过时记录审核人、审核时间和发布时间。
- 驳回必须填写原因，并记录审核人和审核时间。
- 下架必须填写原因，并保留历史活动和场次数据。
- 非法状态转换返回稳定错误码，不依赖异常文本。

## 权限设计

- 公开活动查询允许匿名访问。
- 商家活动接口要求 `MERCHANT` 角色和有效商家归属。
- 管理员审核接口要求 `ADMIN` 角色。
- 商家查询和修改必须限定当前 `merchant_id`。
- 管理员审核不能伪装成商家修改草稿。
- 公开接口只返回 `PUBLISHED` 活动，不暴露审核原因、内部备注和商家员工信息。

新增权限标识：

- `ACTIVITY_MANAGE`
- `ACTIVITY_REVIEW`
- `VENUE_MANAGE`

## 模块边界与文件规模

### 后端

活动领域按职责组织，不创建包含全部逻辑的单一 `ActivityService` 或 `ActivityController`：

```text
com.eventhub.activity
├── api
│   ├── publicview
│   └── merchant
├── application
│   ├── activity
│   ├── venue
│   └── session
├── domain
│   ├── activity
│   ├── venue
│   └── session
├── infrastructure
│   ├── persistence
│   └── cache
└── dto

com.eventhub.admin
├── activity
├── merchant
└── dashboard
```

- Controller 只负责协议转换、参数校验和调用应用服务。
- 应用服务负责用例编排和事务边界，不直接拼装 HTTP 响应。
- 状态转换放入独立状态机或领域策略，不散落在 Controller 和 Mapper 中。
- 商家归属校验使用可复用的上下文服务，所有商家写操作统一调用。
- Mapper 只负责 SQL 和数据映射，公开查询与管理查询按用途拆分。
- Redis 缓存封装在活动详情缓存组件中，业务服务不直接操作 Redis Key。
- DTO、数据库记录和公开响应模型分离，避免数据库字段直接成为公共接口契约。
- 单个 Java 文件应保持单一职责；接近 300 行时必须评估拆分，不以抽取无意义工具类凑行数。
- 多表保存使用明确的聚合事务服务，场馆、活动、场次和审核事务互不混杂。

### 前端

用户端和管理端按功能拆分，不把活动全部写入 `App.tsx`、单个页面或全局 CSS：

```text
src/
├── pages/
├── entities/
│   ├── activity/
│   ├── venue/
│   └── merchant/
├── features/
│   ├── activity-editor/
│   ├── activity-review/
│   ├── activity-search/
│   ├── venue-editor/
│   └── merchant-management/
└── shared/
    ├── api/
    ├── auth/
    ├── ui/
    └── utils/
```

- 页面组件只负责布局、路由参数和组合 Feature。
- API 查询封装在对应 Entity 或 Feature 中，由 TanStack Query 管理服务端状态。
- 表单分步骤拆分，基本信息、场馆、场次票档和提交确认分别维护组件。
- 通用状态标签、分页、金额和日期展示组件放入 `shared/ui` 或对应 Entity。
- 业务数据不写入 Zustand；Zustand 继续只保存登录态和纯客户端状态。
- 每个功能使用独立样式文件，避免继续扩大全局 `app.css`。
- React 页面或组件接近 250 行时评估拆分，复杂表单和表格列定义单独放置。
- OpenAPI 生成目录保持只读，业务适配逻辑放在生成代码之外。

## 前端设计

### 用户端

- 首页从静态精选卡片升级为真实已发布活动。
- 增加活动列表页，支持关键词、分类、城市和日期筛选。
- 增加活动详情页，展示封面、介绍、场馆、场次和票档。
- 固定座位场次展示座位区域、排号和座位等级预览，并明确标记“选座购票即将开放”。
- 未发布、已下架或不存在的活动统一展示不可用状态。
- TanStack Query 管理活动列表和详情数据，不复制到 Zustand。
- 金额使用后端返回的分值格式化展示，不进行浮点业务计算。

### 管理端

- `MERCHANT` 显示活动管理、场馆管理入口。
- `ADMIN` 显示活动审核、商家管理和运营概览入口。
- 活动编辑采用分步表单：基本信息、场馆、场次票档、提交确认。
- 场馆编辑支持无座模式和固定座位模式；固定座位可按区域、排数和每排座位数批量生成后再调整。
- 商家列表展示草稿、待审核、已发布、已驳回和已下架状态。
- 管理员审核页展示活动完整信息、场次票档，并提供通过和填写原因驳回。
- 商家管理页支持创建商家、绑定已有用户和启停商家。
- 工作台使用真实聚合接口展示活动和商家基础指标。
- 商家与管理员继续共用管理端工程，但菜单和路由按角色显示。

## 风险

- 当前活动审核通过即发布，若未来增加定时发布，需要扩展状态和发布时间规则。
- 活动、场次和票档聚合编辑涉及多表事务，必须避免部分保存。
- 批量生成固定座位可能产生较多记录，需要限制单场馆座位数并使用批量插入。
- 商家与管理员共用管理端，菜单隐藏不能替代后端权限和数据归属校验。
- 已发布活动若允许直接修改，会影响后续订单快照，因此本阶段禁止直接修改。
- Redis 缓存删除失败可能短暂返回旧详情；数据库状态校验和较短 TTL 降低影响，后续可用消息补偿。
- 场馆固定座位结构可能较大，本阶段只支持配置，不提前实现高并发座位状态查询。
- 封面 URL 来自外部时可能失效；正式文件管理在 MinIO 阶段解决。

## 回滚方案

- 回滚前后端阶段 2 代码，停止暴露活动相关接口和页面。
- 删除 Redis 中 `eventhub:activity:*` 缓存键。
- 开发环境可在确认无保留数据需求后重建 Docker 数据卷。
- 需要保留数据时通过新的补偿迁移停用或清理阶段 2 表，不修改已发布的 `V2` 文件。
- 回滚不影响阶段 1 用户、角色、登录和 Refresh Token 数据。

## 验收标准

- 商家能够创建场馆、活动、场次和票档并提交审核。
- 管理员能够创建商家、绑定商家员工并控制商家状态。
- 商家能够配置无座场馆或固定座位场馆。
- 管理员能够审核通过、驳回和下架活动。
- 商家不能操作其他商家的数据。
- 普通用户和匿名用户只能看到已发布活动。
- 活动状态不能绕过状态机非法转换。
- 活动详情缓存能正确写入和失效。
- 用户能够查看真实活动、场次、票档及固定座位预览。
- 管理台能够展示来自数据库的基础运营指标。
- 用户端和管理端均通过生成的 API Client 调用后端。
