# EventHub 城市活动预约与票务平台实施文档

## 1. 项目定位

EventHub 是一个面向求职展示的城市活动预约与票务平台。平台支持用户浏览活动、选择场次、锁定座位、创建订单、取消预约和核销入场；同时为商家提供活动发布和订单管理能力，为管理员提供审核、权限和运营管理能力。

项目采用完全前后端分离架构。用户端、管理端和 Spring Boot 后端分别作为独立工程开发、构建、测试和部署，前端只能通过公开的 REST API 与 WebSocket 协议访问后端，Spring Boot 不托管前端页面或静态构建产物。

项目重点不是堆叠中间件，而是通过完整业务链路展示以下后端能力：

- Spring Boot 工程设计与模块划分
- Spring MVC 接口开发与参数校验
- Spring Security 登录认证和权限控制
- MyBatis 数据访问与复杂 SQL
- MySQL 事务、索引和并发控制
- Redis 缓存、分布式锁、限流与幂等
- RabbitMQ 异步消息、重试和死信处理
- WebSocket 实时通知
- Docker Compose 本地部署
- 自动化测试、接口文档、监控和压测

## 2. 实施原则

- 第一版采用模块化单体，不在项目早期引入 Spring Cloud。
- 优先完成可运行的核心业务，再逐步增加 Redis、RabbitMQ 等能力。
- MySQL 是核心业务数据的最终可信来源，Redis 不作为订单主存储。
- 订单、座位、支付等状态必须通过明确的状态机流转。
- 所有关键接口都要考虑权限、事务、幂等、异常和审计。
- 不直接复制开源项目代码，只借鉴工程结构和实现思路。
- 每个阶段都应具备可独立运行、测试和演示的成果。
- 前端与后端禁止通过源码依赖或共享数据库耦合，接口契约统一由 OpenAPI 管理。
- 用户端、管理端和后端使用独立环境变量、构建流程、镜像和发布版本。
- 本地开发允许配置 CORS，生产环境通过 Nginx 提供统一 HTTPS 入口。

## 4. 完全前后端分离架构

### 4.1 工程边界

EventHub 使用三个独立 Git 仓库：

```text
eventhub-server/   Spring Boot 后端
eventhub-web/      用户端 React 应用
eventhub-admin/    商家与平台管理 React 应用
```

三个工程必须满足：

- 独立依赖管理，不共用 `node_modules`、Maven 模块或构建目录。
- 独立执行测试和构建。
- 独立生成版本号和 Docker 镜像。
- 前端不读取后端源码、配置文件或数据库。
- 后端不依赖前端构建产物，也不提供 SPA fallback。
- 用户端与管理端可以按不同发布节奏升级。

如需统一管理部署文件，可额外建立只包含基础设施的仓库：

```text
eventhub-deploy/
├── compose.yaml
├── nginx/
├── prometheus/
└── scripts/
```

### 4.2 运行与通信

开发环境建议地址：

```text
用户端：http://localhost:3000
管理端：http://localhost:3001
后端：http://localhost:8080
REST API：http://localhost:8080/api
WebSocket：ws://localhost:8080/ws
```

通信链路：

```text
eventhub-web / eventhub-admin
        |
        | HTTPS REST + WSS
        v
eventhub-server
        |
        +-- MySQL
        +-- Redis
        +-- RabbitMQ
        +-- MinIO
```

前端只能访问后端暴露的 API、WebSocket 和公开文件地址，不能直接连接 MySQL、Redis、RabbitMQ 或 MinIO 管理端口。

### 4.3 接口契约

- REST API 统一使用 `/api` 前缀。
- 接口使用 JSON，时间字段统一使用 ISO 8601 格式并明确时区。
- OpenAPI 文档由后端代码生成并作为接口契约来源。
- 用户端和管理端根据 OpenAPI 生成 TypeScript 请求类型。
- DTO 变化必须先更新契约，再调整两个前端。
- 响应包含稳定业务错误码，前端不能依赖后端异常文本判断业务。
- 列表接口统一分页模型和排序参数。
- 金额由后端返回字符串或以分为单位的整数，前端禁止自行使用浮点数计算应付金额。

### 4.4 认证方案

推荐采用短期 Access Token 与长期 Refresh Token：

- Access Token 只保存在前端内存状态中，不持久化到 `localStorage`。
- Refresh Token 由后端写入 `HttpOnly`、`Secure` Cookie。
- 前端启动时通过刷新接口恢复登录状态。
- Access Token 失效后，Axios 拦截器只允许发起一次刷新请求，其他失败请求排队等待。
- 刷新失败后清理本地用户状态并跳转登录页。
- 用户端和管理端使用不同的客户端标识与权限范围。
- 后端在 Redis 中保存 Refresh Token 会话，支持退出和强制撤销。

若开发环境为跨站 Cookie，需要正确配置 `SameSite=None; Secure`，这要求使用 HTTPS。为了降低本地环境复杂度，开发阶段优先使用 Vite 代理将 `/api` 和 `/ws` 转发到后端。

### 4.5 CORS 与 CSRF

- 后端使用明确的 Origin 白名单，禁止在携带凭据时配置通配符 `*`。
- 开发环境只允许用户端和管理端的本地地址。
- 生产环境只允许正式用户端和管理端域名。
- Cookie 认证相关接口需要配置 CSRF 防护。
- 普通业务请求使用 `Authorization: Bearer <access-token>`。
- 前端请求携带 Cookie 时显式启用 `withCredentials`。
- WebSocket 握手同样校验 Origin 和访问令牌。

### 4.6 生产部署

建议使用不同域名部署：

```text
https://eventhub.example.com        用户端
https://admin.eventhub.example.com  管理端
https://api.eventhub.example.com    后端 API 与 WebSocket
```

部署职责：

- 用户端和管理端分别构建为静态文件，由各自的 Nginx 或对象存储/CDN 托管。
- Spring Boot 单独构建为 JAR 或容器镜像。
- API 域名只反向代理 Spring Boot，不托管前端文件。
- `/ws` 反向代理必须支持 WebSocket Upgrade。
- 所有外部流量使用 HTTPS/WSS。
- MySQL、Redis 和 RabbitMQ 不暴露到公网。
- 三个工程分别执行 CI/CD，任何一个前端发布都不要求重新构建后端。

## 7. 核心模块

### 7.1 认证与权限

- 用户名或手机号注册
- BCrypt 密码散列
- Access Token 与 Refresh Token
- Spring Security 请求过滤链
- 基于角色和权限标识的接口授权
- 登录失败次数限制
- Refresh Token 撤销和退出登录

第一版可以使用 JWT，但必须支持服务端撤销。推荐在 Redis 中保存会话版本或 Refresh Token，而不是使用完全不可控的无状态 Token。

### 7.2 活动管理

活动状态：

```text
DRAFT -> PENDING_REVIEW -> PUBLISHED -> FINISHED
                        \-> REJECTED
PUBLISHED -> OFF_SHELF
```

主要能力：

- 活动草稿创建和编辑
- 活动分类、标签和场馆
- 场次时间配置
- 固定座位与无座票两种库存模式
- 管理员审核
- 上架、下架和结束

### 7.3 座位锁定

用户选择座位后，系统提供短时间锁定，例如 10 分钟。

基本流程：

```text
校验活动和场次
-> 校验座位可售
-> 获取场次或座位锁
-> 写入 Redis 临时锁定
-> 返回锁定编号和过期时间
-> 用户在有效期内创建订单
```

需要处理：

- 同一座位并发锁定
- 用户重复提交
- 锁定超时释放
- Redis 锁定与 MySQL 售出状态的校验
- 服务重启后的过期订单补偿

### 7.4 订单管理

订单状态：

```text
PENDING_PAYMENT -> PAID -> USED
PENDING_PAYMENT -> CANCELLED
PENDING_PAYMENT -> EXPIRED
PAID -> REFUNDING -> REFUNDED
```

创建订单需要在一个数据库事务中完成：

1. 校验座位锁属于当前用户。
2. 校验活动、场次和票档可售。
3. 锁定或校验数据库库存记录。
4. 创建主订单和订单明细。
5. 将对应座位更新为已售或预占状态。
6. 写入 Outbox 事件。
7. 事务提交后异步发送订单事件。

第一版支付使用模拟支付接口，不接入真实支付平台。

### 7.5 票券与核销

- 支付成功后生成唯一票券码。
- 票券码不能直接使用自增 ID。
- 核销接口仅允许对应商家员工访问。
- 核销需要保证幂等，重复扫描返回已核销结果。
- 核销时间、操作人和设备信息进入审计记录。

### 8.5 关键约束

- 场次座位使用 `(session_id, seat_id)` 唯一索引。
- 票券码建立唯一索引。

- 状态字段使用明确枚举值，禁止使用含义不清晰的魔法数字。

## 9. Redis 设计

建议统一 Key 前缀：

```text
eventhub:{业务}:{标识}
```

| Key 示例 | 数据类型 | 用途 | 建议过期时间 |
| --- | --- | --- | --- |
| `eventhub:auth:refresh:{tokenId}` | String | Refresh Token 会话 | 7 至 30 天 |
| `eventhub:user:profile:{userId}` | String/Hash | 用户资料缓存 | 30 分钟 |
| `eventhub:activity:detail:{activityId}` | String | 活动详情缓存 | 10 分钟 |
| `eventhub:session:stock:{sessionId}` | Hash | 场次票档库存快照 | 按场次结束时间 |
| `eventhub:seat:lock:{sessionId}:{seatId}` | String | 座位临时锁定 | 10 分钟 |
| `eventhub:idempotent:{userId}:{requestId}` | String | 请求幂等 | 10 至 30 分钟 |
| `eventhub:rate-limit:{dimension}` | String | 接口限流 | 按限流窗口 |
| `eventhub:online:{userId}` | String | 在线状态 | 短周期续期 |

缓存策略：

- 查询先读缓存，未命中再读数据库。
- 热点数据设置随机过期时间，减少同时失效。
- 空结果使用短周期缓存，降低缓存穿透。
- 关键活动可使用互斥重建或逻辑过期，降低缓存击穿。
- 数据修改先提交数据库，再删除缓存。
- 缓存删除失败通过重试消息或补偿任务处理。
- 禁止在线上使用 `KEYS` 扫描业务 Key。

## 10. RabbitMQ 设计

### 10.1 建议事件

```text
order.created
order.paid
order.expired
order.cancelled
ticket.generated
ticket.used
activity.published
notification.requested
```

### 10.2 使用场景

- 订单创建后发送延迟消息，用于超时关闭。
- 支付成功后异步生成电子票。
- 订单状态变化后生成站内通知。
- 活动发布后异步刷新搜索索引。
- 核销完成后异步更新统计数据。

### 10.3 可靠性要求

- 生产端使用 Publisher Confirm。
- 数据库事务内写入 Outbox 事件。
- 后台任务投递未发送的 Outbox 事件。
- 消费端使用手动确认。
- 消费逻辑通过消息 ID 保证幂等。
- 失败消息设置有限重试次数。
- 超过重试次数后进入死信队列。
- 提供死信查看和人工重放能力。

- 配置 Java 21 与 Spring Boot 3.5.x。

- 配置代码格式、Checkstyle 或 Spotless。
- 配置统一响应、异常处理和请求 ID。

- 用户端和管理端可以通过代理访问后端健康接口。

### 阶段 1：用户、认证与权限

目标：完成平台身份体系。

任务：

- 创建用户、角色、权限和商家表。
- 使用 Flyway 管理建表脚本。
- 实现注册、登录、刷新和退出。
- 配置 BCrypt。
- 实现 Spring Security 过滤器和异常响应。
- 在 Redis 中管理 Refresh Token 和会话撤销。
- 实现管理员、商家、用户三种角色。
- 用户端实现注册、登录、刷新和退出页面。
- 管理端实现登录页、权限路由和基础后台布局。
- 实现 Axios 单次刷新与失败请求排队机制。
- 添加认证与越权访问测试。

验收：

- 用户能够注册和登录。
- Token 过期后可以刷新。
- 退出后 Refresh Token 立即失效。
- 普通用户不能访问商家和管理员接口。
- 页面刷新后可以恢复登录状态，刷新失败时正确回到登录页。

### 阶段 2：活动、场馆和场次

目标：完成可浏览、可审核的活动数据链路。

任务：

- 创建活动、场馆、场次、票档和座位表。
- 实现商家活动草稿和编辑。
- 实现活动提交审核。
- 实现管理员审核与驳回。
- 实现活动分页、分类和条件查询。
- 为活动详情增加 Redis 缓存。
- 设计并验证活动状态机。
- 用户端实现活动列表、筛选、详情和场次页面。
- 管理端实现活动编辑、场次配置和审核页面。

验收：

- 商家可以创建并提交活动。
- 管理员可以审核。
- 用户只能查看已发布活动。
- 活动修改后缓存能够正确失效。
- 用户端和管理端均通过生成的 API Client 调用后端。

### 阶段 3：座位锁定与订单

目标：完成项目最核心的并发预约链路。

任务：

- 实现场次座位查询。
- 使用 Redis 临时锁定座位。
- 使用 Lua 脚本或 Redisson 保证锁定原子性。
- 实现请求幂等 Key。
- 创建订单主表、明细表和锁定记录表。
- 使用 MySQL 唯一约束防止重复售卖。

- 实现模拟支付和订单取消。
- 用户端实现选座、订单确认、支付结果和订单列表页面。
- 增加并发测试。

验收：

- 两个用户不能同时成功锁定同一座位。
- 重复创建订单不会产生多笔有效订单。
- 过期座位锁可以自动释放。
- 数据库最终不会出现同一场次座位重复售卖。

### 阶段 4：RabbitMQ 与订单超时

目标：完成可靠异步处理。

任务：

- 添加 RabbitMQ 到 Docker Compose。

验收：

- 已支付订单不会被超时消息关闭。
- 重复消费不会生成重复票券。
- RabbitMQ 临时不可用时，Outbox 事件可以补偿发送。

## 13. 测试策略

### 13.1 单元测试

重点覆盖：

- 活动状态转换
- 订单状态转换
- 座位锁定校验
- 权限判断
- 金额计算
- 消息幂等判断

### 13.2 集成测试

使用 Testcontainers 启动真实 MySQL 和 Redis，重点验证：

- MyBatis SQL 和数据库约束
- Flyway 迁移
- Redis 锁定与过期
- 事务回滚
- 订单并发创建

### 13.3 API 测试

重点覆盖：

- 正常请求
- 参数错误
- 未登录和越权
- 重复请求
- 非法状态转换
- 资源不存在
- 限流响应

### 13.4 并发测试

至少设计以下场景：

- 100 个用户争抢同一座位。
- 1000 个请求争抢有限无座票库存。
- 同一用户重复提交创建订单。
- RabbitMQ 重复投递支付成功消息。

核心断言：

- 不超卖。
- 不重复创建有效订单。
- 不重复生成票券。
- 数据库与业务状态最终一致。

### 13.5 前端与端到端测试

- 使用 Vitest 测试请求错误映射、权限判断和金额展示。
- 使用 Testing Library 测试登录、活动筛选和订单状态组件。
- 使用 Playwright 覆盖注册登录、活动浏览、选座下单、模拟支付和票券核销。
- API Mock 只用于前端组件测试，正式端到端测试必须连接真实测试后端。
- 验证无权限菜单不可见，直接访问受限 URL 也会被后端拒绝。

## 14. 安全要求

- 密码只保存 BCrypt 散列。
- Token、密码和验证码不得写入日志。
- 所有写接口执行参数校验。
- 商家接口必须校验数据归属，不能只校验角色。
- 管理接口记录操作日志。
- 上传文件必须校验扩展名、MIME、大小和存储路径。
- 防止 SQL 注入、越权访问、重复提交和暴力登录。
- 对登录、座位锁定、订单创建、核销接口实施限流。
- 生产环境密钥通过环境变量或密钥服务注入。
- 前端生产构建不得包含密码、私钥或后端内部地址。
- 前端展示内容默认转义，富文本内容必须经过白名单清洗。
- 用户端与管理端分别配置 CSP、Referrer-Policy 等安全响应头。

## 15. 可观测性要求

- 每个请求生成并返回 `requestId`。
- 日志包含时间、级别、请求 ID、用户 ID、接口、耗时和错误码。
- 统计登录失败、订单创建、支付成功、订单过期、消息消费失败等指标。
- 健康检查至少覆盖应用、MySQL、Redis 和 RabbitMQ。
- 对慢 SQL、缓存命中率、线程池、JVM 和 MQ 堆积建立监控。
- 前端上报页面加载失败、API 错误和 WebSocket 断线事件，但不得上报 Token 和敏感个人信息。
