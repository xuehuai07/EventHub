# EventHub 版本管理

## 0.4.0 - 2026-06-11

阶段 4“RabbitMQ 与订单异步处理”完成。

### 数据结构

- 新增 Outbox 事件、消息消费记录和电子票表。
- 订单创建和支付事务分别写入 `ORDER_CREATED`、`ORDER_PAID` 事件。
- 为升级前已支付且尚未出票的订单回填支付事件，确保异步电子票能够补生成。
- 保证回填事件的数据库事件编号、JSON 载荷事件编号和消费幂等编号一致。
- 电子票使用随机票号，并通过订单明细和序号唯一约束防止重复生成。
- 将订单明细的场次座位全局唯一索引调整为普通索引，允许取消或过期订单恢复后的座位再次售卖。

### 消息与可靠性

- 新增 RabbitMQ 订单事件、延迟、死信交换机及超时、支付、死信队列。
- 订单超时使用消息 TTL 与死信路由触发，数据库定时扫描继续作为最终兜底。
- Outbox 发布使用持久化消息、Publisher Confirm、不可路由检查和失败退避。
- 消费者在业务事务成功后确认消息，失败有限重试后进入死信队列。
- 消费记录和数据库唯一约束保证重复消息不会重复恢复库存或生成票券。

## 0.3.0 - 2026-06-11

阶段 3“座位锁定与订单”完成。

### 公共接口

- 新增场次实时座位与票档库存查询、座位锁定查询和释放接口。
- 新增用户订单创建、列表、详情、模拟支付和取消接口。
- 新增商家订单与平台管理员订单只读查询接口。
- 用户写接口使用 `Idempotency-Key`，订单创建、支付和取消具备数据库幂等记录。

### 数据结构

- 新增座位锁定、锁定明细、订单、订单明细、支付流水和幂等记录表。
- 固定座位场次按场馆座位等级生成场次座位快照，已有场次通过 Flyway V3 回填。
- 订单状态固定为 `PENDING_PAYMENT`、`PAID`、`CANCELLED`、`EXPIRED`。
- 创建订单时由 MySQL 条件更新完成最终库存占用，取消和超时恢复库存。

### 架构

- 固定座位与无座库存分别使用 Redis Lua 原子锁定。
- 临时锁由 Redis 管理，最终售出状态与库存由 MySQL 约束保障。
- 锁定、订单创建、支付、超时和查询分别位于独立应用服务。
- 用户端完成选座、倒计时、确认、支付和订单查询闭环；管理端提供订单只读视图。

## 0.2.0 - 2026-06-10

阶段 2“活动、场馆与场次”完成。

### 公共接口

- 新增公开活动分类、活动分页筛选和活动详情接口。
- 新增商家场馆、固定座位、活动草稿、场次票档和提交审核接口。
- 新增管理员商家管理、活动审核、驳回、下架和运营概览接口。

### 数据结构

- 新增活动分类、标签、场馆、场馆座位、活动、场次、票档和场次座位表。
- 活动状态固定为 `DRAFT`、`PENDING_REVIEW`、`PUBLISHED`、`REJECTED`、`OFF_SHELF`、`FINISHED`。
- 金额统一使用分为单位的整数，场馆和活动写操作使用商家数据归属。

### 架构

- 活动模块按 API、应用服务、领域规则、持久化和缓存拆分。
- 活动详情使用 Redis 缓存，写操作和状态变化主动失效。
- 用户端和管理端继续通过 OpenAPI 生成客户端访问后端。
- 前端业务页面使用路由懒加载，并按 Entity、Feature、Page 分层。

## 0.1.0 - 2026-06-10

阶段 1“用户认证与权限”完成。

### 公共接口

- 新增 `POST /api/auth/register`、`POST /api/auth/login`。
- 新增 `POST /api/auth/refresh`、`POST /api/auth/logout`。
- 新增 `GET /api/auth/me`。
- 新增 `GET /api/merchant/session`、`GET /api/admin/session`。

### 数据结构

- 新增用户、角色、权限及其关联表。
- 新增商家、商家员工及其关联结构。
- 基础角色固定为 `USER`、`MERCHANT`、`ADMIN`。

### 架构

- Access Token 使用短期 JWT，并仅保存在前端内存中。
- Refresh Token 使用 Redis 会话和 HttpOnly Cookie，支持轮换与撤销。
- 用户端与管理端使用独立客户端类型和 Cookie。
