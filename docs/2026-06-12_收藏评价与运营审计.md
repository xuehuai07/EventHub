# EventHub 收藏、评价与运营审计

## Goal

完成 EventHub 阶段 6“文件、搜索与运营能力”中尚未交付的核心能力：

- 普通用户收藏已发布活动，并查看自己的收藏列表。
- 真实购票用户在对应场次开始后评价活动。
- 用户查看活动评分与公开评价。
- 商家查看本商家的销售概览、趋势和热门活动。
- 平台管理员查看全平台运营概览、趋势和热门活动。
- 平台管理员查询关键管理操作审计记录。

现有本地活动封面上传和活动关键词、分类、城市、日期筛选继续复用，不重复建设。功能交付后，项目稳定版本升级至 `0.7.0`。

## Scope

### 活动收藏

- 已登录的 `USER_WEB` 普通用户可以收藏或取消收藏 `PUBLISHED` 活动。
- 同一用户对同一活动最多保留一条收藏记录。
- 用户可以分页查看自己的收藏活动。
- 活动详情展示收藏总数和当前用户收藏状态。
- 活动下架后不再允许新增收藏，但已存在的收藏仍可在个人列表中显示，并明确活动当前不可售。

### 活动评价

- 用户只有在本人存在该活动的 `PAID` 订单，且对应场次已经开始后才能评价。
- 每个用户对同一活动最多发布一条评价。
- 评价包含 1 至 5 分整数评分和 1 至 1000 字纯文本内容。
- 用户可以更新或删除自己的评价。
- 活动详情展示评分平均值、评价数量和公开评价分页列表。
- 管理员可以隐藏或恢复违规评价；隐藏评价不参与公开列表和评分聚合。
- 评价内容按纯文本处理，不渲染 HTML 或 Markdown。

### 运营统计

- 管理员查看全平台累计成交额、已支付订单数、售出票数、已核销票数、活跃商家数和已发布活动数。
- 商家只查看所属商家的累计成交额、已支付订单数、售出票数、已核销票数和已发布活动数。
- 支持最近 7 天、30 天或指定合法日期范围的成交趋势。
- 支持按成交额或售出票数查询热门活动。
- 统计只基于 MySQL 已支付订单和已生成票券，不使用 Redis 临时库存作为事实来源。
- 管理端工作台根据当前角色调用管理员或商家统计接口，不再由商家调用管理员接口。

### 操作审计

- 记录商家活动、场馆、场次管理，活动提交审核，管理员审核、驳回、下架、商家状态修改和评价隐藏/恢复等关键管理操作。
- 审计记录包含操作人、角色、商家归属、动作、资源类型、资源 ID、结果摘要、请求编号和时间。
- 管理员可以按操作人、动作、资源类型和时间范围分页查询。
- 审计摘要不得记录密码、Token、Cookie、二维码凭证、完整请求头或完整请求体。
- 成功审计记录与对应业务写操作处于同一事务；业务回滚时不保留成功记录。

## Non-goals

- 本阶段不迁移到 MinIO、OSS、COS 或 CDN；现有随机文件名和本地文件存储继续使用。
- 本阶段不引入 Elasticsearch；现有 MySQL 活动筛选继续作为搜索实现。
- 不实现关注商家、收藏夹分组、评价回复、点赞、图片评价或敏感词自动审核。
- 不允许未购票用户评价，也不支持同一用户对同一活动发布多条评价。
- 不实现实时流式统计、数据仓库、复杂用户画像或推荐算法。
- 不增加用户与权限管理页面，不调整现有角色和权限模型。
- 不记录普通读取请求、登录密码、模型请求正文或其他高体量低价值日志。
- 不替代 Actuator、Prometheus 或应用日志等系统可观测性能力。

## Impact

### Database

新增 Flyway V10，创建以下表。

#### `eh_activity_favorite`

| 字段 | 说明 |
| --- | --- |
| `id` | 收藏主键 |
| `user_id` | 收藏用户 |
| `activity_id` | 活动 |
| `created_at` | 收藏时间 |

约束与索引：

- `(user_id, activity_id)` 唯一，防止重复收藏。
- `(user_id, created_at, id)` 用于个人收藏分页。
- `(activity_id)` 用于收藏数量统计。

#### `eh_activity_review`

| 字段 | 说明 |
| --- | --- |
| `id` | 评价主键 |
| `user_id` | 评价用户 |
| `activity_id` | 被评价活动 |
| `rating` | 1 至 5 分整数 |
| `content` | 最长 1000 字纯文本 |
| `status` | `PUBLISHED` 或 `HIDDEN` |
| `hidden_reason` | 管理员隐藏原因 |
| `moderator_id` | 最近操作管理员 |
| `moderated_at` | 最近管理时间 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

约束与索引：

- `(user_id, activity_id)` 唯一，保证每人每活动一条评价。
- `(activity_id, status, created_at, id)` 用于公开评价分页。
- `rating` 使用数据库检查约束限制在 1 至 5。
- 被隐藏评价保留原内容和审计关系，不做物理删除。

#### `eh_operation_log`

| 字段 | 说明 |
| --- | --- |
| `id` | 日志主键 |
| `operator_user_id` | 操作用户 |
| `operator_name` | 操作时的展示名称快照 |
| `operator_role` | `MERCHANT` 或 `ADMIN` |
| `merchant_id` | 商家操作时的商家归属 |
| `action` | 稳定动作编码 |
| `resource_type` | 资源类型 |
| `resource_id` | 资源 ID |
| `summary` | 脱敏后的操作摘要 |
| `request_id` | 请求编号 |
| `created_at` | 操作时间 |

索引：

- `(created_at, id)` 用于全局倒序分页。
- `(operator_user_id, created_at)` 用于操作人筛选。
- `(action, created_at)` 和 `(resource_type, resource_id)` 用于审计定位。

操作日志不建立指向业务资源的外键，避免资源删除或历史演进破坏审计记录。

### Public APIs

活动收藏：

```text
GET    /api/activity-favorites
PUT    /api/activity-favorites/{activityId}
DELETE /api/activity-favorites/{activityId}
GET    /api/activity-favorites/{activityId}/status
```

活动评价：

```text
GET    /api/activities/{activityId}/reviews
GET    /api/activities/{activityId}/review-summary
GET    /api/activities/{activityId}/my-review
PUT    /api/activities/{activityId}/my-review
DELETE /api/activities/{activityId}/my-review
```

管理与统计：

```text
GET  /api/admin/dashboard/operations
GET  /api/admin/dashboard/sales-trend
GET  /api/admin/dashboard/top-activities

GET  /api/merchant/dashboard/operations
GET  /api/merchant/dashboard/sales-trend
GET  /api/merchant/dashboard/top-activities

GET  /api/admin/activity-reviews
POST /api/admin/activity-reviews/{reviewId}/hide
POST /api/admin/activity-reviews/{reviewId}/restore

GET  /api/admin/operation-logs
```

所有新增接口声明稳定 `operationId` 并进入 OpenAPI。列表继续使用项目统一分页响应。

### Existing API Changes

公开活动详情新增以下只读字段：

- `favoriteCount`
- `reviewCount`
- `averageRating`

这属于向后兼容的响应字段扩展。当前用户收藏状态和本人评价通过独立认证接口查询，避免公开活动接口因匿名与登录状态产生不同缓存内容。

### Permissions And Ownership

- 收藏和本人评价接口要求 `USER` 角色，并始终使用当前认证用户 ID。
- 评价资格通过订单和场次 SQL 校验，不接受客户端传入用户 ID、订单 ID 或资格标记。
- 商家统计入口必须校验商家及员工状态，并在 SQL 中使用当前 `merchant_id`。
- 管理员统计、评价管理和操作日志接口要求 `ADMIN`。
- 管理员隐藏评价时必须填写 1 至 500 字原因。

### Cache

- 公开活动详情缓存继续使用 `eventhub:activity:detail:{activityId}`。
- 收藏数量变化、评价创建、更新、删除、隐藏和恢复后删除对应活动详情缓存。
- 当前用户收藏状态和本人评价不进入公共活动详情缓存。
- 统计接口首版直接查询 MySQL，不增加 Redis 缓存；若后续性能数据表明确需要，再单独设计短时缓存。

### Frontend

用户端：

- 活动详情新增收藏按钮、评分摘要和评价区域。
- 未登录用户点击收藏或评价时跳转登录页。
- 不满足评价资格时展示明确原因，不显示可提交表单。
- 新增 `/favorites` 页面展示个人收藏。
- 用户导航增加收藏入口。

管理端：

- 工作台按管理员和商家身份展示不同范围的运营指标。
- 使用图表展示成交趋势，使用表格或排行卡展示热门活动。
- `/audit` 从占位页面替换为管理员操作审计页面；商家不显示该菜单。
- 新增管理员评价治理入口，可合并到审计页面的独立标签中。
- 移除静态日期、阶段 2/5 和旧里程碑文案，改为当前日期与 `0.7.0` 运营视图。

## Implementation Plan

### 1. Database And Domain

- 新增 Flyway V10。
- 新增 `favorite`、`review` 和 `audit` 后端模块或在活动领域内建立清晰子包。
- 收藏写入依赖数据库唯一约束保证幂等。
- 评价资格查询同时校验当前用户、`PAID` 订单、活动归属和场次开始时间。
- 评价聚合只统计 `PUBLISHED` 状态。
- 建立统一 `OperationLogService` 和稳定动作编码。

### 2. Favorites

- 实现收藏、取消收藏、状态和个人分页列表。
- 收藏活动时校验活动当前为 `PUBLISHED`。
- 取消收藏对不存在记录按幂等成功处理。
- 收藏列表返回活动展示快照所需字段及当前活动状态。
- 收藏变更后失效公开活动详情缓存。

### 3. Reviews

- 实现公开评价列表、评分摘要和本人评价 CRUD。
- 更新评价时重新校验活动和当前用户归属，但不要求活动仍处于发布状态。
- 删除本人评价采用物理删除；管理员治理采用 `HIDDEN` 状态保留。
- 管理员隐藏和恢复操作写入操作日志。
- 评价变更后失效公开活动详情缓存。

### 4. Operational Statistics

- 管理员统计 Mapper 查询全平台数据。
- 商家统计 Mapper 始终使用 `merchant_id` 限定。
- 日期范围最大 366 天，默认最近 30 天，按 `Asia/Shanghai` 自然日聚合。
- 成交额和订单数以 `eh_ticket_order.status = 'PAID'` 为准。
- 售出票数以已支付订单的 `total_quantity` 为准；已核销票数以 `eh_ticket.status = 'USED'` 为准。
- 热门活动默认返回前 10 条。

### 5. Audit Logging

- 在活动、场馆、场次、审核、商家状态和评价治理等应用服务的成功事务中显式记录日志。
- 从现有请求 ID 上下文获取 `request_id`；无法获取时允许为空。
- 摘要使用业务字段白名单构建，不序列化完整请求对象。
- 管理员审计查询支持分页和有限筛选，不允许任意字段排序。

### 6. Frontend And OpenAPI

- 重新生成用户端和管理端 OpenAPI 客户端。
- 用户端增加收藏页面、活动详情收藏与评价交互。
- 管理端工作台接入角色化统计接口和趋势图。
- 管理端实现操作审计与评价治理。
- 对新增页面补齐加载、空状态、错误状态和移动端布局。

### 7. Documentation And Versioning

- 更新 `README.md` 的核心能力、常用页面和测试说明。
- 更新 `docs/architecture-interface-requirements.md` 的稳定数据、接口、权限和审计约束。
- 更新 `docs/version-management.md`，新增 `0.7.0`。
- 在本文档的 Implementation Result 中记录实际交付和验证结果。

## Risks

- 公开活动详情加入实时聚合字段会增加查询成本；应使用按活动 ID 的聚合查询，并继续利用详情缓存。
- 评价资格跨订单和场次表查询，必须建立在现有订单索引上并检查执行计划，避免活动详情读取触发资格查询。
- 统计查询的数据量会持续增长；首版限制日期范围和热门数量，后续根据慢 SQL 决定是否增加汇总表。
- 显式审计调用可能遗漏新写操作；本阶段通过动作清单测试覆盖，后续可增加架构测试。
- 操作摘要可能意外包含敏感数据；必须使用字段白名单，不接受任意对象自动 JSON 序列化。
- 管理员隐藏评价会改变评分聚合，需要保证评价状态更新和缓存失效在同一业务流程内。
- 个人收藏列表包含已下架活动时，前端必须禁止进入购票流程并明确展示状态。

## Rollback Plan

- 前端移除收藏、评价、统计和审计入口，不影响现有购票主流程。
- 后端禁用新增接口并回滚应用代码；V10 数据表保留，避免生产数据库执行破坏性降级。
- 公开活动详情新增字段可以保留为默认值，保持旧客户端兼容。
- 审计写入失败必须回滚对应管理操作；若上线后确认影响过大，可在后续版本将审计调整为事务提交后的可靠异步事件。
- 如最终废弃功能，在备份数据后通过新的 Flyway 迁移清理，不修改或删除 V10。

## Verification

### Backend

- 收藏重复提交不产生重复记录，重复取消保持幂等。
- 用户只能读取和修改自己的收藏与评价。
- 未购票、订单未支付、其他用户订单或场次未开始时不能评价。
- 同一用户同一活动只能存在一条评价。
- 隐藏评价不出现在公开列表，也不参与评分数量和平均分。
- 商家统计不能读取其他商家数据。
- 管理员和商家趋势数据与订单、票券事实数据一致。
- 操作日志与成功业务事务一致，失败或回滚操作不写成功日志。
- 审计摘要不包含密码、Token、Cookie、二维码凭证或完整请求正文。
- 运行 `eventhub-server/mvnw.cmd clean verify`。

- 活动收藏状态、收藏数量和收藏列表在增删后及时刷新。
- 匿名用户收藏或评价会进入登录流程。
- 符合资格的用户可以创建、更新和删除评价。
- 管理员和商家工作台展示各自范围数据，不发生 `403`。
- 趋势图、热门活动、审计筛选和评价治理具备加载、空状态和错误状态。
- 移动端活动详情和收藏页无横向滚动。
- 运行用户端和管理端 `npm run check`。

## Acceptance Criteria

- 用户可以稳定收藏和取消收藏已发布活动，并查看个人收藏列表。
- 只有真实购票且场次已开始的用户可以评价活动。
- 活动详情展示准确的收藏数、评分和公开评价。
- 管理员可以隐藏和恢复违规评价，公开评分同步更新。
- 商家只能查看本商家销售统计，管理员可以查看全平台统计。
- 管理端不再展示静态旧阶段工作台，管理员和商家均可正常加载自己的运营视图。
- 关键管理操作具有可查询、脱敏且事务一致的审计记录。
- 后端、两个前端、OpenAPI、Flyway 和跨账号数据隔离验收全部通过。
- README、架构约束和版本记录更新到 `0.7.0`。
