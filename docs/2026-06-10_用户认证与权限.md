# EventHub 阶段 1：用户认证与权限

## 目标

建立平台身份体系，完成普通用户、商家和管理员三类身份的注册、登录、会话恢复、令牌刷新、退出及接口授权，为后续活动、订单和核销模块提供稳定的用户上下文。

## 范围

- 使用 Flyway 创建用户、角色、权限、用户角色、角色权限、商家和商家员工表。
- 支持用户名或手机号作为登录标识，密码使用 BCrypt 散列。
- 实现注册、登录、刷新、退出和当前用户接口。
- 使用短期 JWT Access Token 和 Redis Refresh Token 会话。
- Refresh Token 使用 HttpOnly Cookie，支持轮换、撤销和退出后立即失效。
- 对 Refresh Token Cookie 写接口增加 CSRF 校验。
- 实现普通用户、商家、管理员的角色和权限授权。
- 对登录失败进行基于登录标识和 IP 的限流。
- 用户端实现注册、登录、当前用户状态和退出。
- 管理端实现登录、权限路由、会话恢复和退出。
- Axios 实现单次刷新请求与并发失败请求排队。
- 由 OpenAPI 重新生成两个前端的 TypeScript 客户端。
- 添加认证、令牌刷新、退出和越权访问测试。

## 非目标

- 本阶段不接入短信验证码、第三方登录、实名认证或找回密码。
- 不实现商家认证材料提交和审核流程，只建立商家及员工身份基础结构。
- 不实现完整用户、角色和权限管理页面。
- 不实现活动、订单、支付和票券业务。
- 不引入 OAuth2 授权服务器或 Spring Authorization Server。
- 不把 Access Token 保存到 `localStorage`、`sessionStorage` 或 Cookie。

## 影响

### 数据库

新增以下表：

- `eh_user`
- `eh_role`
- `eh_permission`
- `eh_user_role`
- `eh_role_permission`
- `eh_merchant`
- `eh_merchant_staff`

核心约束：

- 用户名和手机号分别建立唯一索引，允许未填写的一项为 `NULL`。
- 用户至少提供用户名或手机号之一。
- 角色编码固定为 `USER`、`MERCHANT`、`ADMIN`。
- 用户、商家和员工均使用明确状态字段。
- 关联表使用联合唯一索引防止重复授权。
- 业务表包含创建时间和更新时间。

### Redis

- `eventhub:auth:refresh:{tokenId}`：Refresh Token 会话，默认 14 天。
- `eventhub:auth:login-failure:{identifier}:{ip}`：登录失败计数，默认 10 分钟。
- `eventhub:auth:login-lock:{identifier}:{ip}`：短期登录锁定，默认 15 分钟。

Redis 只保存 Refresh Token 的不可逆摘要、用户编号、客户端类型、过期时间和 CSRF 摘要，不保存明文密码或完整令牌。

### 公共 API

```text
POST /api/auth/register
POST /api/auth/login
POST /api/auth/refresh
POST /api/auth/logout
GET  /api/auth/me
GET  /api/merchant/session
GET  /api/admin/session
```

`/api/merchant/session` 和 `/api/admin/session` 用于验证管理端身份边界，并为后续管理端模块提供当前操作人上下文。

### 前端

- 用户端增加登录页、注册页和登录用户区域。
- 管理端增加登录页、受保护布局和角色路由。
- 两个前端增加内存认证状态。
- 页面启动时通过 Refresh Token 恢复会话。
- Access Token 失效时只发起一次刷新，其余请求等待刷新结果。

## 安全设计

### Access Token

- 使用 JWT，默认有效期 15 分钟。
- 使用环境变量提供的密钥签名，生产密钥不得使用仓库默认值。
- 包含用户编号、角色、权限、客户端类型和会话编号。
- 前端只保存在内存状态中。

### Refresh Token

- 使用密码学安全随机数生成，不使用可长期独立验证的 JWT。
- Cookie 设置 `HttpOnly`、`SameSite=Lax`，生产环境启用 `Secure`。
- 用户端与管理端使用不同 Cookie 名，防止客户端会话相互覆盖。
- 每次刷新后轮换 Refresh Token，旧 Token 立即删除。
- 退出时删除 Redis 会话和对应 Cookie。
- 服务端仅保存 Token 摘要，数据库和日志中不保存明文 Token。

### CSRF

- 登录成功时同时下发可读取的 CSRF Cookie。
- 刷新和退出请求必须携带 `X-CSRF-Token`。
- 服务端校验请求头、Cookie 和 Redis 会话中的 CSRF 摘要。

### 登录保护

- 同一登录标识和 IP 在 10 分钟内连续失败 5 次后锁定 15 分钟。
- 登录成功后清除失败计数。
- 错误响应不区分“用户不存在”和“密码错误”，避免账号枚举。
- 密码和 Token 不写入日志。

### 客户端边界

- `USER_WEB`：只允许普通用户登录。
- `ADMIN_WEB`：只允许商家员工或管理员登录。
- 普通用户访问商家或管理员接口返回 `403`。
- 商家访问管理员接口返回 `403`。

## 风险

- Cookie、CSRF、Vite 代理和跨域配置不一致会导致刷新失败。
- Refresh Token 轮换若缺少并发保护，可能造成重复刷新或误判盗用。
- JWT 密钥配置错误会导致所有 Access Token 失效。
- Flyway 已执行的迁移不能直接修改，后续调整必须新增迁移。
- 手机号注册暂不验证手机号所有权，只适合作为 MVP 身份入口。
- 管理端同时承载商家和管理员，需要严格区分角色和后续数据归属。

## 回滚方案

- 代码回滚到阶段 0，停止暴露认证接口。
- 删除 Redis 中 `eventhub:auth:*` 会话键，使现有会话全部失效。
- 开发环境可执行 `docker compose down -v` 后重新创建空数据库。
- 若需要保留其他数据，先备份数据库，再通过独立回滚脚本删除阶段 1 新增表。
- 已发布的 Flyway 文件不修改；正式环境通过新的补偿迁移处理结构调整。
- 前端回滚后不再读取认证 Cookie，后端退出接口仍可用于清理旧会话。

### 集成验收

- 注册普通用户并登录用户端。
- 刷新页面后自动恢复登录状态。
- 模拟 Access Token 过期后自动刷新。
- 多个并发 `401` 只触发一次刷新请求。
- 退出后刷新页面保持未登录。
- 普通用户不能登录管理端。
- 商家可以登录管理端但不能访问管理员接口。
- 管理员可以访问管理员接口。
- Redis 中可观察到 Refresh Token 会话，退出后立即删除。

## 验收标准

- 用户能够通过用户名或手机号注册和登录。
- 密码只以 BCrypt 散列形式保存。
- Access Token 过期后可以安全刷新。
- Refresh Token 每次使用后轮换，旧 Token 不能再次使用。
- 退出后 Refresh Token 立即失效。
- 普通用户不能访问商家和管理员接口。
- 商家不能访问管理员接口。
- 页面刷新后可以恢复登录状态，恢复失败时回到登录页。
- 两个前端不持久化 Access Token。
- 所有自动检查和集成验收通过。
