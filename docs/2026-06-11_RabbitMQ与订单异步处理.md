# EventHub 阶段 4：RabbitMQ 与订单异步处理

## 目标

在现有座位锁定和订单事务基础上引入 RabbitMQ 与 Outbox，完成可靠的订单超时关闭和支付后电子票异步生成。RabbitMQ 负责及时触发，MySQL 保存最终状态和可补偿事件，现有数据库超时扫描继续作为兜底。

## 范围

- 在 Docker Compose 中增加 RabbitMQ 管理版服务和健康检查。
- 后端接入 Spring AMQP、发布确认、不可路由消息返回和事务完成后的消费确认。
- 新增 Outbox 事件表、消息消费记录表和最小电子票表。
- 订单创建事务内写入 `ORDER_CREATED` Outbox 事件。
- 支付成功事务内写入 `ORDER_PAID` Outbox 事件。
- 定时发布未发送或待重试的 Outbox 事件。
- 使用消息 TTL 与死信交换机触发订单超时关闭。
- 支付成功后异步生成一人一票的电子票记录。
- 消费者通过事件编号和数据库唯一约束保证幂等。
- 消费失败有限重试，超过次数后进入死信队列。
- 保留现有订单超时数据库扫描任务，作为消息丢失、延迟或 RabbitMQ 不可用时的最终补偿。

## 非目标

- 不接入真实支付渠道、退款或退款状态。
- 不提供票券列表、详情、二维码或核销 API。
- 不实现票券核销、操作审计、站内通知或 WebSocket 推送。
- 不增加 RabbitMQ 管理页面的业务代理接口。
- 不引入 RabbitMQ 延迟交换机插件，避免依赖非标准镜像和插件生命周期。
- 不删除现有数据库超时扫描补偿。
- 不修改现有用户端或管理端公开业务接口。

## 影响

### 数据库

新增 Flyway `V4` 迁移，创建：

- `eh_outbox_event`
  - `event_id` 为全局唯一事件编号。
  - 保存聚合类型、聚合编号、事件类型、JSON 载荷、发布状态、重试次数、下次重试时间和最后错误。
  - 状态使用 `PENDING`、`PUBLISHED`、`FAILED`。
  - 按状态和下次重试时间建立扫描索引。
- `eh_message_consume_record`
  - 使用 `(consumer_name, event_id)` 唯一约束记录消费完成。
  - 消费副作用和完成记录必须处于同一数据库事务。
- `eh_ticket`
  - 每张票关联订单、订单明细、用户、活动和场次。
  - 无座票档按订单明细数量生成多张票，固定座位订单每个订单明细生成一张票。
  - 使用 `(order_item_id, unit_no)` 唯一约束避免重复生成。
  - 票号使用不可预测随机值并建立唯一索引。
  - 本阶段状态只使用 `UNUSED`，阶段 5 再扩展核销状态和核销信息。

不修改已经执行的 `V1`、`V2`、`V3` 迁移。

### 订单事务

订单创建事务在订单、明细、幂等记录成功保存后写入 `ORDER_CREATED` Outbox 事件。事件载荷至少包含：

```json
{
  "eventId": "UUID",
  "orderId": 1,
  "orderNo": "EH...",
  "paymentDeadlineAt": "2026-06-11T12:00:00"
}
```

模拟支付事务在订单状态改为 `PAID` 且支付记录写入后，写入 `ORDER_PAID` Outbox 事件。事件载荷至少包含事件编号、订单编号和支付时间。

Outbox 与订单业务数据使用同一 MySQL 事务，禁止在订单事务提交前直接向 RabbitMQ 发布消息。

### RabbitMQ 拓扑

使用持久化交换机和队列：

```text
eventhub.order.events       topic exchange
eventhub.order.delay        direct exchange
eventhub.order.dlx          topic exchange

eventhub.order.timeout.delay.q
eventhub.order.timeout.q
eventhub.order.paid.q
eventhub.order.dead.q
```

路由：

```text
ORDER_CREATED
-> eventhub.order.delay / order.timeout.delay
-> 按 paymentDeadlineAt 设置消息 expiration
-> eventhub.order.timeout.delay.q
-> TTL 到期后死信到 eventhub.order.events / order.timeout
-> eventhub.order.timeout.q

ORDER_PAID
-> eventhub.order.events / order.paid
-> eventhub.order.paid.q

消费失败超过重试上限
-> eventhub.order.dlx / order.timeout.dead 或 order.paid.dead
-> eventhub.order.dead.q
```

订单超时消息的 TTL 使用 `max(paymentDeadlineAt - 当前时间, 0)`。若 RabbitMQ 暂时不可用，Outbox 后续补发时不会重新获得完整 15 分钟等待时间。

标准 RabbitMQ 的单队列消息 TTL 可能受队首消息影响。订单创建时间和支付截止时间在当前业务中基本同序，数据库扫描任务仍负责修正极端延迟。

### Outbox 发布

- 定时任务按 `next_attempt_at` 扫描 `PENDING` 和 `FAILED` 事件。
- 发布消息使用持久化消息、Publisher Confirm 和 mandatory 路由检查。
- 只有收到 Broker 确认后才将事件更新为 `PUBLISHED`。
- 发布失败记录简化后的错误原因，增加重试次数并按退避时间更新 `next_attempt_at`。
- 多实例可能重复发布同一事件，消费者必须始终按至少一次投递处理。
- 不在数据库事务中长时间等待 RabbitMQ 确认。

### 消费幂等

订单超时消费者：

1. 校验事件结构。
2. 查询消费记录，已完成则直接确认。
3. 在事务内按 `PENDING_PAYMENT` 条件更新订单为 `EXPIRED`。
4. 只有状态更新成功时恢复库存。
5. 已支付、已取消或已过期订单不重复改变状态或库存。
6. 写入消费完成记录，事务提交后确认消息。

支付事件消费者：

1. 校验订单当前状态为 `PAID`。
2. 按订单明细和数量生成电子票。
3. 依赖 `(order_item_id, unit_no)` 唯一约束防止重复票。
4. 写入消费完成记录，事务提交后确认消息。

消费者由监听容器在业务事务成功返回后 ACK。处理失败时进行有限本地重试，超过上限后拒绝且不重新入队，由队列死信配置转入统一死信队列。

## 风险

- RabbitMQ 与数据库不能组成单一事务，必须依靠 Outbox 和消费幂等达到最终一致。
- Publisher Confirm 成功但数据库状态更新失败会导致重复发布，消费者唯一约束必须覆盖全部副作用。
- 标准 TTL 死信队列存在队首阻塞可能，数据库扫描兜底不可删除。
- 消费者在库存恢复后、消费记录提交前失败会回滚整个数据库事务；库存恢复和订单状态更新必须使用同一事务。
- RabbitMQ 长时间不可用会增加 Outbox 积压，需要索引、批量上限和退避避免数据库压力。
- 死信队列只保留失败消息，不自动修复业务，需要日志和数据库记录支持排查。
- 最小电子票表会成为阶段 5 的稳定数据基础，字段和唯一约束需要保守设计。

## 回滚计划

- 停止 Outbox 发布任务和 RabbitMQ 消费监听。
- 恢复仅使用数据库超时扫描关闭订单，订单和库存最终状态不依赖 RabbitMQ。
- 保留 `eh_outbox_event`、`eh_message_consume_record` 和 `eh_ticket` 数据，不删除已经执行的 Flyway 迁移。
- 回滚应用代码后，未发布 Outbox 事件不会影响现有订单接口。
- RabbitMQ 服务可从 Compose 停止，但不自动删除持久化卷。
- 若电子票生成逻辑异常，暂停 `order.paid` 消费者并通过唯一约束保护已有票记录，修复后重新投递。

## 验证

### 自动化测试

- 订单创建与 Outbox 事件在同一事务成功或回滚。
- 支付成功与 `ORDER_PAID` 事件在同一事务成功或回滚。
- 同一超时事件重复消费只恢复一次库存。
- 已支付订单收到超时事件后状态和库存不变。
- 同一支付事件重复消费不生成重复电子票。
- Outbox 发布失败后保持可重试，确认成功后标记已发布。
- 消费失败超过上限后进入死信队列。

### 集成验收

1. 启动 MySQL、Redis 和 RabbitMQ，确认三者健康。
2. 创建支付截止时间较短的测试订单，确认到期后由消息触发为 `EXPIRED`。
3. 创建并支付订单，等待超过支付截止时间，确认仍为 `PAID`。
4. 重复投递同一 `ORDER_PAID` 事件，确认票券数量和订单数量一致且不增加。
5. 停止 RabbitMQ 后创建订单，确认 Outbox 保留待发送事件。
6. 恢复 RabbitMQ，确认事件补发且过期订单最终关闭。
7. 模拟消费者持续失败，确认消息进入死信队列。
8. 运行后端测试、两个前端检查和 `git diff --check`。

## 验收标准

- RabbitMQ、MySQL 和 Redis 在 Docker Compose 中均通过健康检查。
- 订单事务不直接依赖 RabbitMQ 可用性。
- 未支付订单到期后优先由消息关闭，数据库扫描可完成兜底。
- 已支付订单不会被延迟超时消息关闭。
- RabbitMQ 临时不可用时 Outbox 事件可在恢复后补发。
- 重复消息不会重复恢复库存或重复生成电子票。
- 消费失败有有限重试和可观察的死信落点。
- 不改变现有订单 API 和前端购买流程。
- 后端、用户端、管理端检查及真实基础设施验收全部通过。

## 实施结果

已于 2026 年 6 月 11 日完成：

- Docker Compose 已增加 RabbitMQ 4.1 管理版服务、持久化卷和健康检查。
- Flyway V4 已创建 Outbox、消息消费记录和电子票表。
- Flyway V5 已将订单明细的场次座位唯一索引调整为普通索引，修复取消或过期订单恢复座位后无法再次购买的问题。
- Flyway V6 已为阶段 4 上线前已支付且尚未出票的订单补写 `ORDER_PAID` 事件。
- Flyway V7 已统一回填事件的数据库事件编号、JSON 载荷事件编号和消费幂等编号。
- 订单创建和支付事务已分别写入 `ORDER_CREATED`、`ORDER_PAID` Outbox 事件。
- Outbox 发布器已实现持久化消息、Publisher Confirm、不可路由检查、错误记录和指数退避。
- 已声明订单事件、延迟和死信交换机，以及超时延迟、超时消费、支付消费和统一死信队列。
- 未支付订单通过消息 TTL 和死信路由触发关闭，数据库扫描继续以 60 秒默认间隔兜底。
- 支付成功后按订单明细数量异步生成随机票号电子票。
- 消费记录和票券唯一约束已验证重复支付事件不会重复出票。
- 消费失败经过 3 次本地重试后进入死信队列。
- RabbitMQ 停止期间订单仍可创建，Outbox 记录失败并在 RabbitMQ 恢复后补发。
- 真实验收中，支付订单保持 `PAID` 并生成票券，未支付订单在截止时间后变为 `EXPIRED` 且座位恢复。
- 后端自动化测试增至 21 项并全部通过。
