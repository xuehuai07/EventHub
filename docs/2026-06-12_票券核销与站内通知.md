# EventHub 阶段 5：票券核销与站内通知

## Goal

打通支付后的履约闭环：用户支付后可以查看电子票和动态二维码，商家可以扫码或输入票码核销本商家票券，系统保证票券只能首次成功核销一次，并通过站内通知和 WebSocket 向用户同步订单与出票状态。

## Scope

### 电子票

- 用户可以分页查看自己的电子票。
- 用户可以查看单张票券详情，包括活动、场次、场馆、票档、座位、订单和核销状态。
- 用户可以从已支付订单详情进入对应票券。
- 票券状态扩展为：
  - `UNUSED`
  - `USED`
  - `CANCELLED`
- 票券记录核销时间、核销商家员工和核销设备摘要。
- 订单继续保持现有状态模型，不增加订单级 `USED`。多人票订单允许部分票券已核销、部分未核销。

### 动态二维码

- 用户端为未核销票券展示短时有效的签名二维码凭证。
- 二维码凭证至少包含票券编号、用户编号、签发时间、过期时间和随机编号。
- 二维码使用独立 HMAC 密钥签名，默认有效期 60 秒。
- 用户端在凭证到期前自动刷新二维码。
- 永久票号仍可用于人工输入核销，但页面不把数据库自增主键作为票码。
- 服务端核销时同时支持动态二维码凭证和人工输入的永久票号。

### 商家核销

- 管理端将“票券核销”占位页替换为真实核销工作台。
- 支持摄像头扫码和手工输入票码。
- 商家只能查询和核销归属于自己商家的票券。
- 平台管理员可以查询核销结果，但本阶段不允许代替商家核销。
- 核销使用 MySQL 条件更新完成 `UNUSED -> USED`。
- 并发提交同一票券时只有一个请求返回首次核销成功。
- 重复扫描已核销票券返回已有核销信息，不重复改变状态。
- 核销页面展示活动、场次、场馆、票档、座位、持票人和核销时间。

### 核销审计

- 新增票券核销记录表。
- 记录票券、商家、操作人、核销结果、设备摘要、请求 IP 和时间。
- 至少记录 `SUCCESS` 和 `ALREADY_USED`。
- 审计记录只保存必要请求元数据，不保存 Access Token、二维码签名密钥或完整请求头。
- 商家可以查看本商家的近期核销记录。

### 站内通知

- 新增站内通知表和用户通知接口。
- 通知类型至少包括：
  - `ORDER_PAID`
  - `ORDER_CANCELLED`
  - `ORDER_EXPIRED`
  - `TICKET_ISSUED`
- 支持通知分页、未读数量、单条已读和全部已读。
- 通知创建与对应业务状态变化处于同一 MySQL 事务。
- 用户端增加通知入口、未读角标和通知列表。

### WebSocket

- 后端增加 STOMP WebSocket 端点 `/ws`。
- STOMP `CONNECT` 必须携带 Access Token，服务端验证后绑定用户身份。
- 用户只订阅自己的 `/user/queue/notifications` 和 `/user/queue/status`。
- 订单支付、取消、超时关闭和票券生成事务提交后推送状态事件。
- 用户端收到事件后刷新订单、票券、通知和未读数量查询。
- WebSocket 只用于及时通知，页面初始数据和断线恢复仍通过 HTTP 查询，MySQL 仍是最终可信来源。

## Non-goals

- 不接入真实支付、退款或退票。
- 不实现管理员通用操作审计页面；本阶段只提供票券核销审计。
- 不发送短信、邮件、微信模板消息或移动端推送。
- 不实现离线核销、弱网本地票库或多设备数据同步。
- 不实现票券转赠、实名制持票人变更或分票。
- 不允许商家撤销已完成核销。
- 不实现活动开始前的定时提醒任务。
- 不使用订单状态表达部分或全部核销结果。

## Risks

- 永久票号二维码容易被截图长期复用，因此二维码必须使用短时签名凭证；人工票号仅作为现场兜底。
- 商家归属只在前端隐藏不足以防越权，核销查询和更新必须在 SQL 或事务内校验 `merchant_id`。
- 先查询后更新会产生重复核销竞态，首次成功必须依赖 `WHERE status = 'UNUSED'` 条件更新。
- WebSocket 连接不能信任浏览器页面状态，STOMP `CONNECT` 必须独立验证 Access Token。
- WebSocket 推送可能丢失或断线，客户端必须在重连后重新查询 HTTP 数据。
- 通知推送不能与业务数据库组成分布式事务，通知记录是最终依据，推送失败只影响及时性。
- 摄像头能力受 HTTPS、浏览器权限和设备限制，必须保留手工输入。
- 现有票券已保存订单明细快照，但场次开始时间来自场次表；已产生订单的场次继续禁止修改。
- 核销后不支持撤销，错误核销需要后续人工补偿流程，本阶段不提供自动回滚。

## Rollback plan

- 停止 WebSocket 推送和前端实时连接，HTTP 票券与通知查询仍可使用。
- 隐藏管理端核销入口并停止核销写接口。
- 保留 V8 新增字段、核销记录和通知数据，不回滚已执行的 Flyway 迁移。
- 已核销票券不自动恢复为 `UNUSED`，避免造成重复入场。
- 回滚应用代码后，原有订单、异步出票和活动流程不受影响。

## Verification

### 自动化测试

- 用户只能查询自己的票券和通知。
- 商家只能预览和核销本商家的票券。
- 管理员不能调用商家核销写接口。
- 有效动态凭证可核销，篡改或过期凭证被拒绝。
- 同一票券并发核销只有一次首次成功。
- 重复核销返回相同核销时间和操作人，不重复更新票券。
- 核销成功与成功审计记录处于同一事务。
- 支付、取消、超时和出票创建正确通知。
- 重复支付事件不生成重复票券或重复出票通知。
- 通知已读接口不能操作其他用户通知。
- WebSocket 连接缺少或携带无效 Token 时被拒绝。

### 集成验收

1. 支付订单并等待 RabbitMQ 异步出票。
2. 用户票券列表出现与订单数量一致的票券。
3. 打开票券详情并验证二维码按 TTL 刷新。
4. 商家扫描二维码并完成首次核销。
5. 同时提交多次核销，确认只有一次首次成功。
6. 再次扫描返回已核销信息。
7. 使用其他商家账号核销，确认不可查看或操作。
8. 验证支付和出票通知、未读数量、已读状态。
9. 断开 WebSocket 后完成业务操作，重新连接并通过 HTTP 查询恢复最新状态。
10. 运行后端、两个前端和 `git diff --check`。

## Acceptance criteria

- 用户支付后可以在用户端查看全部电子票。
- 未核销票券可以展示短时动态二维码。
- 商家可以通过摄像头扫码或手工票码核销所属活动票券。
- 同一票券在并发和重复请求下只能首次成功一次。
- 商家无法查看或核销其他商家的票券。
- 每次成功或重复核销都有可查询的审计记录。
- 用户可以查看站内通知、未读数量并标记已读。
- 支付、取消、超时和出票状态通过 WebSocket 及时触发页面刷新。
- WebSocket 断线不影响业务正确性，HTTP 查询可以恢复最终状态。
- 后端、用户端、管理端自动检查和真实基础设施验收全部通过。
